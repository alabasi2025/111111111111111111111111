# وثيقة تحليل المتطلبات غير الوظيفية (NFR)

**المهمة 3: تحليل المتطلبات غير الوظيفية - أنشئ وثيقة تحتوي على متطلبات الأداء، الأمان، قابلية التوسع، والتوافقية**

تهدف هذه الوثيقة إلى تحديد المتطلبات غير الوظيفية (Non-Functional Requirements - NFRs) للنظام المقترح، والتي تصف كيفية عمل النظام بدلاً من وظائفه المحددة. هذه المتطلبات ضرورية لضمان جودة النظام، وكفاءته، وأمنه، وقدرته على النمو المستقبلي.

---

## 1. متطلبات الأداء (Performance Requirements)

تتعلق متطلبات الأداء بمدى سرعة واستجابة النظام تحت عبء عمل معين. يجب أن يضمن النظام تجربة مستخدم سلسة وفعالة.

| المقياس | الوصف | الهدف |
| :--- | :--- | :--- |
| **زمن الاستجابة (Response Time)** | يجب ألا يتجاوز زمن استجابة 95% من المعاملات الحرجة (مثل تسجيل الدخول، حفظ قيد محاسبي) حدًا معينًا. | **< 2 ثانية** |
| **الإنتاجية (Throughput)** | يجب أن يكون النظام قادرًا على معالجة عدد محدد من المعاملات في الثانية. | **500 معاملة/ثانية** كحد أدنى |
| **التحميل المتزامن (Concurrent Users)** | يجب أن يدعم النظام عددًا محددًا من المستخدمين النشطين في نفس الوقت دون تدهور في الأداء. | **10,000 مستخدم نشط** |
| **استخدام الموارد (Resource Utilization)** | يجب ألا يتجاوز استخدام وحدة المعالجة المركزية (CPU) والذاكرة (RAM) حدًا معينًا تحت عبء العمل الأقصى. | **< 70%** |
| **زمن تحميل الصفحة (Page Load Time)** | يجب أن يتم تحميل جميع واجهات المستخدم (UI) بشكل كامل. | **< 1.5 ثانية** |

---

## 2. متطلبات الأمان (Security Requirements)

يجب أن يطبق النظام أعلى معايير الأمان لحماية البيانات الحساسة وضمان سلامة النظام، مع دمج تقنيات الذكاء الاصطناعي لتعزيز الأمن السيبراني [1] [2].

| المقياس | الوصف | المنهجية |
| :--- | :--- | :--- |
| **التحقق من الهوية (Authentication)** | يجب دعم التحقق متعدد العوامل (MFA) والتحقق البيومتري (Biometric Authentication) للوصول إلى النظام [3]. | **OAuth 2.0، OpenID Connect، دعم FIDO2/WebAuthn** |
| **الترخيص (Authorization)** | يجب تطبيق نموذج التحكم في الوصول القائم على الدور (RBAC) مع صلاحيات دقيقة على مستوى الوحدة والميزة والوحدة الفرعية [4]. | **صلاحيات هرمية دقيقة (وحدة ← مؤسسة ← فرع)** |
| **حماية البيانات (Data Protection)** | يجب تشفير جميع البيانات الحساسة (مثل كلمات المرور، البيانات المالية) أثناء النقل (In Transit) وأثناء التخزين (At Rest). | **TLS 1.3، تشفير AES-256** |
| **الوصول المقيد (Restricted Access)** | يجب تقييد الوصول عبر المتصفح بوجود اتصال آمن يتم إنشاؤه بواسطة الذكاء الاصطناعي، وتطبيق آلية التعرف على الأجهزة المصرح لها [5]. | **آلية التعرف على الأجهزة (Device Recognition) وربط المستخدمين بها** |
| **التحكم الزمني (Time-based Access)** | يجب تطبيق التحكم في الوصول بناءً على وقت الاستخدام، مع آلية لطلب الوصول خارج الأوقات المحددة [3]. | **تحديد أوقات عمل محددة لكل مستخدم/دور** |
| **التدقيق والمراجعة (Auditing)** | يجب تسجيل جميع الإجراءات الحرجة (مثل التعديل، الحذف، تنزيل البيانات) وتوثيقها، وربط صلاحيات التعديل والحذف بصلاحيات التتبع والمراجعة [6]. | **سجل تدقيق (Audit Log) غير قابل للتغيير** |
| **مكافحة الاحتيال (Anti-Fraud)** | يجب تطبيق آلية التحقق من الترخيص عبر الإنترنت (Online License Verification) للحد من السرقة والاحتيال [4]. | **نظام أكواد تفعيل مركزي عبر الإنترنت** |

## 3. متطلبات قابلية التوسع (Scalability Requirements)

يجب تصميم النظام باستخدام أحدث التقنيات لضمان قابليته للتوسع الأفقي والعمودي لدعم نمو الشركات الكبيرة والمؤسسات [7].

| المقياس | الوصف | المنهجية |
| :--- | :--- | :--- |
| **التوسع الأفقي (Horizontal Scaling)** | يجب أن تكون البنية التحتية قادرة على إضافة المزيد من الخوادم (Servers) وقواعد البيانات (Databases) بسهولة لتوزيع الحمل. | **بنية خدمات مصغرة (Microservices Architecture)** |
| **التعامل مع البيانات (Data Handling)** | يجب أن يدعم النظام نموًا كبيرًا في حجم البيانات وعدد المعاملات دون تدهور في الأداء. | **استخدام قواعد بيانات NoSQL/SQL حديثة وموزعة** |
| **قابلية التكوين (Configurability)** | يجب أن تكون البنية الأساسية قابلة للتثبيت والتفعيل بشكل منفصل لكل واجهة وميزة ونظام [4]. | **فصل الوحدات (Modules) وإدارة الصلاحيات على مستوى الميزة** |
| **دعم المؤسسات المتعددة (Multi-Tenancy)** | يجب أن يدعم النظام إنشاء عدة مؤسسات منفصلة تمامًا داخل الوحدة الواحدة [8]. | **هيكل هرمي (الوحدة ← المؤسسات ← الفروع)** |
| **التكنولوجيا الأساسية (Core Technology)** | يجب استخدام أحدث وأعلى التقنيات المتاحة لضمان قابلية التوسع والنمو المستقبلي [7]. | **اختيار أطر عمل (Frameworks) ولغات برمجة حديثة ومثبتة الكفاءة** |

## 4. متطلبات التوافقية (Compatibility Requirements)

تتعلق متطلبات التوافقية بقدرة النظام على العمل بشكل صحيح مع بيئات وأنظمة مختلفة.

| المقياس | الوصف | الهدف |
| :--- | :--- | :--- |
| **المتصفحات (Browsers)** | يجب أن يكون النظام متوافقًا بشكل كامل مع أحدث إصدارات المتصفحات الرئيسية. | **Chrome، Firefox، Edge، Safari (آخر إصدارين)** |
| **الأجهزة (Devices)** | يجب أن يكون النظام مستجيبًا (Responsive) ويعمل بكفاءة على مختلف أحجام الشاشات والأجهزة. | **الحواسيب المكتبية، الأجهزة اللوحية، الهواتف الذكية** |
| **نظام التشغيل (Operating System)** | يجب أن يعمل النظام بشكل مستقل عن نظام التشغيل الخاص بالعميل. | **العمل عبر الويب (Web-based) بالكامل** |
| **التكامل (Integration)** | يجب أن يدعم النظام واجهات برمجة تطبيقات (APIs) موحدة لتمكين التكامل مع أنظمة خارجية (مثل أنظمة الدفع، أنظمة الرسائل). | **RESTful APIs، دعم Webhooks** |
| **التنبيهات (Notifications)** | يجب دعم إرسال التنبيهات والإشعارات عبر قنوات متعددة يحددها المستخدم [9]. | **واتساب، البريد الإلكتروني، إشعارات داخل التطبيق** |

## المراجع

[1] دعم الذكاء الاصطناعي للأمن السيبراني وحماية الخادم.
[2] تفضيل البحث واستنساخ جميع مميزات الرقابة والأمان من الأنظمة الذكية العالمية.
[3] منهجية التحكم في الوصول بناءً على وقت الاستخدام (Time-based Access Control) وآلية طلب الوصول خارج الأوقات المحددة وتعزيزها بالتحقق البيومتري (Biometric Authentication).
[4] منهجية تفعيل الميزات والواجهات بشكل منفصل بناءً على الصلاحيات وأكواد التفعيل والتحقق من الترخيص عبر الإنترنت وفصل واجهات المطور والإدارة والمستخدم.
[5] تقييد الوصول عبر المتصفح بوجود اتصال آمن ينشئه الذكاء الاصطناعي وآلية التعرف على الأجهزة المصرح لها وربط المستخدمين بها لمنع تسرب البيانات.
[6] ربط صلاحيات التعديل والحذف بصلاحيات التتبع والمراجعة لضمان التدقيق الأمني.
[7] تفضيل استخدام أحدث التقنيات لضمان قابلية النظام للتوسع (Scalability) للشركات الكبيرة وتطبيق أعلى المعايير.
[8] منهجية الهيكل الهرمي للنظام (الوحدة ← المؤسسات ← الفروع) وإمكانية إنشاء مؤسسات متعددة ومنفصلة داخل الوحدة الواحدة وواجهة إدارة العمليات البينية.
[9] تفضيل إرسال التنبيهات والإشعارات عبر قنوات متعددة يحددها المستخدم (مثل واتساب والبريد الإلكتروني).
